# Debugger Tab

The **Debugger** tab is a fully interactive, source-level CAOS debugger. It consists of a persistent **agent list panel** on the left and a detailed **source view and inspector** on the right. 

![Debugger Tab](/docs/media/tab_debugger.png)

## Agent List Panel

The left panel displays all agents that currently have running scripts. It is designed to keep agents organized and easy to track over time.

* **Classifier Grouping:** Agents are intelligently grouped by their script's Family/Genus/Species (e.g., all `2 13 9` scripts are grouped together).
* **Stable Sorting:** Within each group, agents are sorted by ID and maintain fixed positions. Incremental DOM updates ensure the list doesn't flicker during auto-refresh.
* **Gallery Names:** Agents display their sprite base name (e.g., `balloonbug`, `lift`, `norn`) for quick visual identification.
* **Event Labels:** Agent badges and group headers display the human-readable CAOS event name alongside the event number (e.g., `evt 9 (Timer)`).
* **State Dots:** Colour-coded indicators represent the agent's current script state:
  * **Green:** Running
  * **Orange:** Blocking / Waiting
  * **Orange Outline:** Paused at a breakpoint
  * **Grey Outline:** Finished execution
* **Stale Agents:** Agents whose scripts have completed are greyed out and persist in the list for 5 seconds before being removed, allowing you to trace recently finished scripts.
* **Search:** Filter the agent list by ID, gallery name, or classifier using the integrated search box.
* **Creature Priority:** Groups containing creature agents (Family 4: Norns, Grendels, Ettins) are always sorted to the top of the list for easy access.

## Source View

Selecting an agent populates the main **Source View** with its script.

* **Formatted Source:** The full CAOS source code is displayed with line numbers. It is auto-formatted with line breaks and 4-space indentation for readability.
* **Execution Highlight:** The current execution position (IP) is highlighted in orange.
* **Syntax Highlighting:** Keywords, numbers, and strings are colour-coded.
* **Follow Toggle:** A checkbox in the toolbar allows you to enable or disable auto-scrolling. When enabled, the view snaps to the current execution line as the script runs. Disable this to freely browse the source code while the script is active.

## Breakpoints & Stepping

Breakpoints can be set directly within the Source View.

* **Toggling Breakpoints:** Click any line number to toggle a breakpoint (indicated by a red marker in the gutter).
* **Execution Pause:** When the engine reaches a breakpoint, the script pauses *before* executing the instruction at that line.
* **Step Controls:** Once paused, use the toolbar controls to navigate execution:
  * **Continue:** Resume execution until the next breakpoint is hit or the script completes.
  * **Step (Into):** Execute exactly one CAOS instruction and pause again.
  * **Step Over:** Execute the next instruction, but skip over nested blocks (such as loops or subroutines).

*Note: Breakpoints can also be managed via the [CAOS IDE](tab_caos_ide.md).*

## Camera Focus

The **Focus** button in the toolbar instantly moves the in-game camera to center on the currently selected agent's position in the world, utilizing the `cmrp` CAOS command.

## Inspector Panel

The right-hand **Inspector Panel** provides real-time visibility into the internal state of the Virtual Machine for the selected agent.

* **Context Variables:** Displays the current values of `OWNR`, `TARG`, `FROM`, `IT`, and the bytecode `IP`.
* **State Badge:** Shows whether the script is `RUNNING`, `BLOCKING`, `PAUSED`, or `FINISHED`.
* **OV Variables:** Displays the current values of any non-zero Object Variables (`OV00`–`OV99`) for the agent.
* **VA Variables:** When the script is *paused* at a breakpoint, this section displays any non-zero Local Script Variables (`VA00`–`VA99`).
* **Breakpoint List:** Lists all active breakpoints with convenient remove buttons.

## Technical Limitations

* Breakpoints are set against bytecode IP addresses (character offsets into the source). The mapping from source lines to IPs relies on the `DebugInfo` address map generated by the compiler.
* Stepping operates at the bytecode instruction level, not necessarily the CAOS source statement level.
* Breakpoints are specific to individual agents, not global across the scriptorium. (Use the CAOS IDE to bind breakpoints across multiple agents).
